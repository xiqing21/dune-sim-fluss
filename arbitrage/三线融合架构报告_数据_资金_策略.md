# 三线融合架构：数据流转 · 资金流转 · 策略流转

## 从 FlinkCDC → Fluss → Flink → Paimon 到量化交易的端到端闭环

---

## 报告摘要

本文以"数据流转、资金流转、策略流转"三条主线重新组织 FlinkCDC → Fluss → Flink → Paimon 架构在 Web3 量化交易场景的完整闭环。数据流转解决"信息从哪里来、怎么实时融合"，资金流转解决"钱在不同市场间怎么对冲、怎么流转、风控怎么保障"，策略流转解决"信号怎么生成、怎么验证、怎么执行"。三条线交汇于 Fluss 统一流数据总线，形成"数据驱动策略 → 策略驱动资金 → 资金产生新数据"的飞轮效应。

## 一、数据流转线：多源实时融合

### 1.1 数据从哪里来

量化交易的数据需求不是单一数据源能满足的。ForthTech 作为持牌量化对冲基金（香港4号+9号牌照），以中性套利为核心策略体系——包括复合套利、配对交易、DEX-CEX跨平台套利、做市和流动性提供，至少需要四类数据：

**链上数据（DEX 侧）**：Uniswap/PancakeSwap 的 Swap 事件、Pool 创建事件、流动性变化事件。这些是链上公开的，但获取速度和深度是关键——L2 每 200ms-2s 产出区块，传统索引器（Dune）分钟级延迟根本跟不上。

**CEX 数据（订单簿侧）**：Binance/OKX/Gate 的 WebSocket 深度流、交易流、资金费率、持仓量、清算流。这是量化策略的核心输入——价差计算需要毫秒级的盘口数据，资金费率决定持仓成本，清算流预示市场情绪突变。

**预言机数据（验证侧）**：Chainlink Data Feeds（链上共识价格）、Data Streams（亚秒级签名报告）、Proof of Reserve（储备金验证）。这不是替代 CEX 数据，而是为价格提供去中心化验证层——当 CEX 报价与 Chainlink 签名报告偏差超过 1% 时，本身就是重要的交易信号。

**私有数据（策略侧）**：内部交易信号、地址标签库、历史回测结果、持仓快照。这些是量化团队的 Alpha 来源，不能与公开数据混存。

### 1.2 数据怎么融合

当前市场没有平台在底层架构层面完整融合这四类数据。Dune 只有链上数据，Nansen 只有链上+标签数据，Amberdata 有链上+CEX 但面向机构合规而非交易执行。核心瓶颈不是"数据有没有"，而是"数据怎么统一时间、统一格式、统一查询"。

Fluss 的统一数据总线架构解决了这个融合问题：

```
┌──────────────────────────────────────────────────────────────────┐
│                    Fluss 统一流数据总线                            │
│                                                                  │
│  fluss://onchain/          链上 DEX Swap、Pool 事件、治理投票      │
│    dex_swaps (PK Table)    主键: (chain_id, block_number, tx_hash)│
│    pool_events (PK Table)  实时更新储备量                         │
│    nft_events (PK Table)   NFT 铸造/交易                         │
│                                                                  │
│  fluss://cex/              CEX 深度、交易、费率、清算              │
│    depth_updates (Log)     Binance/OKX/Gate 增量深度流            │
│    orderbook_l2 (PK Table) 实时重建的 L2 订单簿                   │
│    funding_rates (PK Table) 资金费率实时更新                       │
│    liquidations (Log)      清算事件流                             │
│                                                                  │
│  fluss://oracle/           Chainlink 预言机验证数据                │
│    chainlink_feeds (PK)    链上共识价格                           │
│    chainlink_streams (Log) 亚秒级签名报告                         │
│    chainlink_por (PK)      储备金验证                             │
│                                                                  │
│  fluss://private/          私有策略数据                           │
│    trading_signals (PK)    内部交易信号                           │
│    address_labels (PK)     地址标签库                             │
│    backtest_results (PK)   回测结果                               │
│    portfolio_snapshots(PK) 持仓快照                               │
│                                                                  │
│  Tiering Service (30s 间隔) → Paimon 数据湖                      │
└──────────────────────────────────────────────────────────────────┘
```

**融合的关键不是"存在一起"，而是"能实时 Join"**。传统架构（Kafka + 独立数据库）中，CEX 价格在 Kafka，链上事件在另一个数据库，要做跨源 Join 只能在应用层手动拼接。Fluss + Flink 的 Delta Join 将多源关联下沉到存储层——链上 Swap 事件与 CEX 深度流的关联无需 TB 级 Flink 状态，通过 Fluss 点查实现零状态 Join，资源消耗降低 10 倍。

## 二、资金流转线：对冲与风控

### 2.1 资金在哪里流转

量化交易的资金流转不是"买入卖出"这么简单。以 Spread 套利为例，资金在以下市场间流动：

**CEX 现货市场**：Gate/Bitget 买入现货 RAVE，资金从 USDT → RAVE。
**CEX 合约市场**：Binance/OKX 开空 RAVE 永续，缴纳保证金（USDT），资金从 USDT → 保证金。
**资金费率流转**：每 8 小时（或 1 小时），合约多头付给空头（正 Funding）或反向（负 Funding）。
**清算流转**：保证金率低于维持率时，交易所强制平仓，资金从保证金 → 对手方。
**链上流转**：大户从链上钱包转入 CEX（预示抛压），或从 CEX 提出到链上（预示囤积）。

### 2.2 资金流转的风控节点

每个流转节点都有风控需求，而 Fluss + Flink 的实时能力正好覆盖：

| 风控节点 | 风险 | Fluss + Flink 方案 |
|---------|------|-------------------|
| 开仓时刻 | Spread 不够覆盖滑点+手续费 | Flink 实时计算净 Spread = 粗价差 - 手续费 - 预估滑点 - Gas（链上执行时） |
| 持仓期间 | 资金费率累计成本过高 | Flink 滑动窗口累计 Funding 支出，超过阈值自动预警 |
| 持仓期间 | 保证金率逼近维持率 | Flink Keyed State 实时维护每个仓位的保证金率 |
| 极端行情 | ADL 强制减仓 | Flink 监控交易所 ADL 指标，排名靠前时自动减仓 |
| 跨市场 | 链上大户转入 CEX 预示砸盘 | Flink Interval Join：链上 Transfer 后 5 分钟内 CEX 充值关联 |
| 预言机 | Chainlink 喂价偏差导致错误清算 | Flink CEP：3 次价格突变在 30 秒内 + 大额借贷 → 操纵预警 |

### 2.3 资金流转的闭环

```
资金入金: USDT → CEX 账户
    │
    ▼
开仓: 现货买入 + 合约开空
    │
    ├──→ Fluss cex.orderbook_l2 (实时保证金率)
    ├──→ Fluss cex.funding_rates (累计 Funding 成本)
    ├──→ Fluss onchain.transfers (链上大户动向)
    │
    ▼
Flink 实时风控引擎:
    - 保证金率 < 150% → 预警
    - Funding 累计 > 阈值 → 评估是否继续持仓
    - 链上大额转入 CEX → 预警砸盘风险
    - Spread 收窄到目标 → 触发平仓
    │
    ▼
平仓: 现货卖出 + 合约平空
    │
    ▼
资金出金: USDT ← CEX 账户
    │
    ▼
Fluss analytics.trade_pnl (记录盈亏, 归档到 Paimon)
```

**关键洞察**：传统量化工具（FMZ、Hummingbot）的风控是"策略层"的——用户在策略代码中自行处理。Fluss 架构将风控下沉到"数据层"——多源数据统一汇入后，Flink 做实时风控计算，任何策略都可以订阅同一个风控信号流。这意味着风控逻辑不需要每个策略重写一遍。

## 三、策略流转线：信号生成 → 验证 → 执行

### 3.1 策略从哪里来

量化策略不是凭空产生的，而是数据驱动的。以下是从数据到策略的流转路径：

**策略类型一：Spread 套利（高确定性）**

信号来源：Fluss cex.depth_updates 与 Fluss onchain.dex_swaps 的 Delta Join → 实时价差计算。
触发条件：净 Spread > 阈值（扣除手续费、滑点、预估 Funding 成本）。
执行路径：CCXT 统一 API → 同时在 Gate 买入现货 + Binance 开空合约。
风控保障：Flink Keyed State 实时跟踪持仓保证金率和 Funding 累计。

**策略类型二：Funding Rate 套利（中确定性）**

信号来源：Fluss cex.funding_rates + Fluss cex.liquidations → 极端 Funding + 清算潮。
触发条件：年化 Funding 超过阈值 + OI 暴增 + 现货薄盘。
执行路径：现货多 + 合约空，持仓吃 Funding。
风控保障：Flink 滑动窗口累计 Funding 收益 vs 价格波动风险。

**策略类型三：链上 Alpha 信号（低确定性高回报）**

信号来源：Fluss onchain.transfers + Fluss private.address_labels → 大户地址资金流向。
触发条件：标记为 Smart Money 的地址大额转入 CEX + 对应币种 Spread 异常。
执行路径：跟单操作或预警通知。
风控保障：Flink Interval Join 检测链上转账后 CEX 充值关联，确认资金流向。

**策略类型四：预言机偏差套利（新类型）**

信号来源：Fluss onchain.dex_swaps + Fluss oracle.chainlink_streams + Fluss cex.depth_updates → 三源交叉验证。
触发条件：DEX 价格 vs CEX 报价 vs Chainlink 签名报告，三者偏差超过 1%。
执行路径：偏差本身是交易信号——Chainlink 喂价滞后时，可以抢在预言机更新前执行。
风控保障：Flink CEP 检测预言机操纵模式（30 秒内 3 次价格突变 + 大额借贷），5 秒预警。

### 3.2 策略怎么验证

传统验证方式是离线回测——用历史数据跑策略看 PNL。但这种方式有两个根本缺陷：回测与实盘偏差率可达 20-30%（Lambda 架构下流批数据不一致），历史数据可能不包含 CEX 订单簿深度（Dune 根本没有 CEX 数据）。

Fluss + Paimon 的 Union Read 解决了这个问题：

```
策略验证流程:

1. 历史回测:
   SELECT * FROM dex_swaps        -- Fluss 热数据(6h) + Paimon 冷数据(全量)
   WHERE token = 'RAVE'
     AND ts BETWEEN '2026-01-01' AND '2026-03-31'
   -- Union Read 自动合并两层, 一个 SQL 获取完整历史

2. CEX 数据关联:
   SELECT s.price as dex_price, c.price as cex_price, 
          s.price - c.price as spread
   FROM dex_swaps s
   INNER JOIN cex.depth_updates c  -- Delta Join, 零状态
   ON s.token = c.symbol
     AND s.ts BETWEEN c.ts - INTERVAL '5' SECOND AND c.ts

3. 策略模拟:
   Flink 作业回放历史数据流 → 策略逻辑实时计算 → PNL 曲线输出
   -- 同一个 Flink SQL, 回测和实盘完全一致, 消除流批偏差
```

### 3.3 策略怎么执行

```
策略流转闭环:

数据层 → 信号层 → 决策层 → 执行层 → 反馈层
  │          │          │          │          │
  ▼          ▼          ▼          ▼          ▼
Fluss     Flink       规则引擎    CCXT      Fluss
多源汇入  信号生成    风控校验    统一API   执行结果
          Delta Join  仓位检查    下单执行   回写归档
          CEP检测     资金检查    状态跟踪   PNL计算
          窗口聚合    策略优先级  异常熔断   绩效评估
```

**策略执行的延迟瓶颈分析**：

数据→信号：Flink Delta Join < 200ms（Fluss 点查替代 TB 级状态）
信号→决策：规则引擎 < 10ms（纯内存计算）
决策→执行：CCXT API → 交易所 < 200ms（网络延迟为主）
**端到端**：数据到执行 < 500ms

对于 Spread 套利来说，500ms 的端到端延迟是足够的——因为 Spread 的持续时间是分钟到小时级别（不是微秒级 HFT）。真正需要微秒级延迟的场景是做市和抢跑，那不是 Spread 套利的赛道。

## 四、三线交汇：飞轮效应

三条线不是独立的，而是形成正反馈飞轮：

```
                数据流转
               ↗        ↖
              /            \
    策略驱动 ←              → 数据驱动
    资金流转                  策略流转
              \            /
               ↘        ↙
                实时闭环
```

**飞轮机制**：

1. 数据流转 → 策略流转：多源实时数据融合生成 Alpha 信号（链上大户转入 CEX + Spread 异常 → 提前布局对冲）
2. 策略流转 → 资金流转：策略信号驱动资金在不同市场间对冲执行（Spread 套利开仓、Funding 收割持仓）
3. 资金流转 → 数据流转：执行结果产生新数据（交易记录回写 Fluss → Tiering 到 Paimon → 丰富历史回测数据 → 优化策略参数）

**飞轮加速器——AI 向量搜索**：

Flink `ML_PREDICT` 对链上事件文本描述生成 Embedding → 写入 Milvus/Fluss 向量列 → `VECTOR_SEARCH` 实现"以事件找事件"的语义搜索。当新的市场异常发生时，毫秒级检索历史相似行情及其收益/风险，为策略决策提供参考。这是所有现有量化平台都不具备的能力。

### 5.3 ForthTech 现有架构的痛点推测

ForthTech 自研了四大平台（数据系统、交易系统、风控系统、策略系统），技术栈 Python/C++/Rust，正在招聘 HFT 方向的 Quant Developer。与微软 Azure 和 OpenAI 深度合作，正在 SaaS 化输出（Azure Marketplace 一键部署）。Darren 同时运营 OnePort（定位"数字资产的恒生电子系统"）。

基于以上信息推测，ForthTech 在快速发展中可能面临的架构痛点：

- **风控逻辑重复开发**：每个策略都要自己算保证金率、监控 Funding、处理 ADL，尚未统一到数据层
- **数据管道碎片化**：四大平台之间数据共享可能仍然依赖 API 调用，而非统一数据总线
- **回测与实盘偏差**：历史数据与实时数据格式和延迟不同
- **链上迁移的数据层缺口**：策略和资金正在迁移到链上，但链上数据基础设施尚未跟上
- **HFT 与中性套利的延迟分层**：正在招 HFT 方向，但现有架构可能未做延迟分层

三线架构解决的核心问题就是：**将"策略层融合"升级为"数据层融合"，风控和数据管道统一，策略只管逻辑**。同时通过三层架构适配不同延迟需求——HFT 走极速层（C++/Rust），中性套利走实时层。

## 六、关键问题解答：Fluss 架构是否太"后分析型"

### 6.1 Fluss 的延迟特性

Fluss 不是传统意义上的"后分析型"系统。它的列式流存储设计目标是**实时分析**，不是 T+1 报表：

- **热数据读取延迟**：<1ms（本地磁盘）
- **客户端缓存后读取延迟**：亚毫秒级
- **Flink 作业端到端延迟**：毫秒级（Fluss 读取 < 1ms + Flink 计算 < 200ms）

与 Kafka 的延迟对比：Kafka 的消费延迟也在毫秒级，但 Kafka 无法做列裁剪、无法做主键点查、无法做 Delta Join——这些在量化场景中意味着要么多传 80% 无用数据，要么维护 TB 级 Join 状态。

### 6.2 量化场景的延迟需求分层

不是所有量化策略都需要微秒级延迟：

| 策略类型 | 延迟需求 | Fluss 适配性 |
|---------|---------|-------------|
| Spread 套利 | 秒级（价差持续分钟级） | 完全适配 |
| Funding Rate 套利 | 分钟级（Funding 每 8h 结算） | 完全适配 |
| 链上 Alpha 信号 | 秒级（链上事件 12s 区块） | 完全适配 |
| 做市策略 | 毫秒级 | 不适配（需要 FPGA/Co-location） |
| 抢跑/Sniper | 微秒级 | 不适配（需要 Mempool 监控+直连） |

ForthTech 的主力策略（Spread + Funding + DEX 套利）全部在秒级延迟范围内，Fluss 架构完全适配。

### 6.3 如果需要更快怎么办

对于需要更低延迟的核心交易路径，可以在 Fluss 架构之上叠加一层**极速决策层**：

```
极速层（交易执行路径）:
  CEX WebSocket → 内存规则引擎 → CCXT API → 交易所
  延迟: < 50ms
  不经过 Fluss/Flink，直接内存处理

实时分析层（风控+信号路径）:
  CEX WebSocket → Fluss → Flink → 信号/风控
  延迟: < 500ms
  提供 Delta Join、CEP、窗口计算等能力

历史分析层（回测+研究路径）:
  Fluss → Tiering → Paimon → Trino/StarRocks
  延迟: 秒级到分钟级
  提供全量历史查询和复杂分析
```

这种分层架构让交易执行走极速路径（< 50ms），风控和信号走实时路径（< 500ms），研究和回测走历史路径。三层共享 Fluss 统一数据总线，数据一致性由 Tiering Service + Union Read 保障。

## 七、总结

三线融合架构的核心价值不是"替代"现有量化工具（FMZ、Hummingbot），而是**升级数据基础设施**——从"策略层融合"到"数据层融合"，从"每策略独立风控"到"统一风控信号流"，从"回测与实盘割裂"到"一套 Flink SQL 跑回测和实盘"。

对于 ForthTech 来说，这意味着：

- 数据团队只需维护一套 Fluss 数据管道（替代 Kafka + 多个独立数据库）
- 策略团队只需关注策略逻辑（风控、数据融合由 Flink + Fluss 统一处理）
- HFT 方向可以走极速层（C++/Rust 内存引擎），中性套利走实时层，各取所需
- 与微软 Azure + OpenAI SaaS 化方向完全互补——Lakehouse 架构天然支持云化输出
- 链上迁移的数据基础设施得到保障——链上事件实时摄入、与 CEX 数据融合、执行结果回写
- TradFi 客户看到的是熟悉的 Lakehouse 架构（FlinkCDC → Fluss → Paimon = 实时版的数据中台）
- 年化可避免损失在数千万美元级别（DeFi 清算预警 + 预言机监控 + 跨链桥安全）
