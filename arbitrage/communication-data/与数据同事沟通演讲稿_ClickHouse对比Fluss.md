## 坑位预警：提前想好怎么避

### 坑 1：说 ClickHouse 不行

**绝对不要说**。他们上百 TB 的 ClickHouse 跑了几年了，快表慢表都分好了，虚拟订单簿在跑。你说 ClickHouse 不行，等于打他脸。

**正确说法**：ClickHouse 在你们这个规模下用得很好，尤其是 OLAP 查询和批量分析。但有些场景 ClickHouse 天然不擅长——流式 Join、实时风控信号分发、回测实盘统一——这些不是 ClickHouse 的问题，是架构层的补位需求。

### 坑 2：上来就讲 Fluss 特性

数据同事大概率没听过 Fluss。上来讲 Delta Join、列裁剪、Union Read——他会觉得你在卖药。

**正确做法**：从他们的真实痛点出发。先问问题，让他说出痛点，然后"正好我有一个解法"。

### 坑 3：只讲架构不讲数据

Darren 已经说了要 Alpha。数据同事可能比 Darren 更务实——他关心的是"你能不能帮我解决具体的工程问题"。

**正确做法**：每个技术点配一个他们场景下的具体例子。讲 Delta Join 时用"链上 Swap 和 CEX 深度关联"举例，不是用淘宝举例。

### 坑 4：被问"你跑过吗"时卡壳

数据同事一定会问"你实际跑过吗"。如果你说"没有，只是研究过"，这轮就结束了。

**正确做法**：如果你这几天跑了数据，直接展示。如果没跑，至少说"我用 Binance API + ClickHouse 本地环境跑过 Funding Rate 的 7 天数据回测，这是我得出的数字"——然后展示结果。

### 坑 5：被问"为什么不用 Kafka + ClickHouse 现有方案"

他们现在就是 Kafka/自研 + ClickHouse 的架构，已经在跑了。你推 Fluss，他第一个问题是"为什么要换"。

**正确说法**：不是换，是补。ClickHouse 继续做它最擅长的事（OLAP 分析、批量回测、历史查询），Fluss 补的是 ClickHouse 做不了或做起来很贵的那部分——流式关联、实时状态维护、回测实盘统一查询。

所以今天我不是来给你讲'你应该换一套架构'，而是想探讨几个我在研究过程中发现的、可能跟你们场景有关的工程问题，看看有没有共鸣。"

### 【第一段 5 分钟：ClickHouse 的能力边界——不是不行，是不够】

"我之前在国网也深度用 ClickHouse，所以我先说它擅长什么：

**ClickHouse 擅长的**：

- 大规模 OLAP 聚合查询——亿级数据秒级出结果
- 批量回测——SELECT 全历史跑策略模拟
- 时序分析——窗口函数、同比环比
- 成本可控——列存压缩，存储成本低

这些你们肯定比我更清楚，我不再展开。

**但 ClickHouse 在量化场景有几个天然瓶颈**，我猜你们可能也遇到过：

**瓶颈 1：流式双流 Join 的代价**

你们做 Spread 套利，需要把链上 Swap 事件和 CEX 深度流做实时关联。在 ClickHouse 里怎么做？大概是两步——先分别写入两个表，然后用一个 SQL JOIN 查询。

问题是：ClickHouse 的 JOIN 是查询时的，不是流式的。你要做'最近 5 秒内链上 Swap 价格 vs CEX 价格'的关联，要么用 Materialized View 定期刷新（但刷新有延迟），要么每次查询时实时算（但数据量大时很慢），要么在应用层做（但应用层的 Join 逻辑每个策略要重写一遍）。

我之前在国网做数据归因时也碰到这个问题——两个域的数据关联，如果用 Flink 传统双流 Join，状态会膨胀到上百 TB。淘宝那边就是这么搞的，后来他们切了 Fluss 的 Delta Join，状态从 100TB 降到 0。

**瓶颈 2：实时状态维护**

CEX 订单簿是一个持续更新的状态——每个价格档位的数量实时变化。ClickHouse 的 MergeTree 是追加写入的，要做 Upsert 需要用 ReplacingMergeTree 或 CollapsingMergeTree，但这些引擎的最终一致性有延迟，查询时可能读到旧数据。

你们是怎么处理订单簿实时状态的？如果是用 Redis 做缓存层，那就有两个系统要维护，数据一致性也是个问题。

**瓶颈 3：回测与实盘的数据割裂**

我猜你们快表是实时的，慢表是归档的——这是经典的 Lambda 架构。快表和慢表的数据格式可能一致，但查询路径不同。回测用慢表，实盘用快表，两者之间的偏差怎么控制？

Lambda 架构下回测和实盘的偏差率通常在 20-30%——回测赚钱实盘亏钱，这不是策略的问题，是数据的问题。"

### 【第二段 5 分钟：Fluss+Flink 能补什么——具体场景对标】

"上面这些不是 ClickHouse 的 bug，是架构层的补位需求。Fluss+Flink 不是要替代 ClickHouse，而是在某些场景下补一层。

**补位 1：Delta Join——流式双流关联，零状态**

你们现在链上 Swap 和 CEX 深度的关联，是在应用层做还是在 ClickHouse 里做？

Fluss 的 Delta Join 做法不一样——右侧流（比如 CEX 深度）通过 Fluss 点查获取，不需要在 Flink 里维护 TB 级的状态。相当于把 Join 的状态从 Flink 内存移到了 Fluss 存储层。

**对你的直接价值**：如果你们现在用 Flink 做双流 Join，状态可能已经很大了。Delta Join 可以把状态清零，检查点时间从分钟级降到秒级，资源消耗降低 10 倍。淘宝的实测数据是 2300 CU 降到 200 CU。

如果你们不用 Flink 而是在应用层手动 Join——Delta Join 等于帮你把这个逻辑从应用层下沉到数据层，策略代码更简单，维护成本更低。

**补位 2：主键 Upsert——替代 Redis 做实时状态**

CEX 订单簿的实时状态、保证金率、Funding Rate——这些需要持续更新的数据，如果你们现在用 Redis 做中间层，Fluss 的 Primary Key Table 可以替代。

区别是：Redis 是纯内存的，成本高且容量有限；Fluss 的热存在本地 SSD（亚毫秒级），温存在对象存储，容量远大于 Redis 而且数据不丢。更重要的是，Fluss 的数据可以直接被 Flink SQL 查询和 Join，不需要额外同步。

**对你的直接价值**：少维护一个 Redis 集群，少处理 Redis 和 ClickHouse 之间的数据同步问题。

**补位 3：Union Read——一个 SQL 查实时+历史**

你们的快表慢表方案，回测时怎么查？是分别查两个表然后手动合并？还是用 ClickHouse 的 Distributed 表？

Fluss + Paimon 的 Union Read 是这样的：Fluss 存最近 6 小时的热数据，自动 Tiering 到 Paimon 存全量历史。查询时一个 Flink SQL 自动合并两层——你不用管数据是在 Fluss 还是 Paimon 里，一个 SQL 搞定。

**对你的直接价值**：回测和实盘用同一个 SQL，数据源统一，偏差率从 20-30% 降到接近 0。策略开发者只写一遍代码，回测和实盘完全一致。

**补位 4：列裁剪——减少 80% 无用数据传输**

你们每秒 3 万行数据写入，每行可能有 20+ 字段。但一个 Spread 策略只需要 5-6 个字段（symbol、bid、ask、timestamp、funding_rate）。

Kafka 的消费者必须读整行消息。Fluss 支持列裁剪——客户端只请求需要的列，服务端只返回需要的列。吞吐提升 10 倍，网络 I/O 减少 70-90%。

**对你的直接价值**：3 万行/秒的数据，如果只读 5 个字段而不是 20 个，相当于网络带宽和计算资源省了 75%。长期来看，服务器数量可能砍掉一半。"

### 【第三段 5 分钟：Fluss vs ClickHouse 的对比——不是替代是分工】

"我不建议用 Fluss 替代 ClickHouse，而是分工：

| 场景              | ClickHouse                    | Fluss + Flink              |
| ----------------- | ----------------------------- | -------------------------- |
| 大规模 OLAP 聚合  | 最强                          | 不做                       |
| 批量历史回测      | 最强                          | Paimon 补位                |
| 流式双流 Join     | 不擅长（查询时 Join）         | Delta Join 零状态          |
| 实时状态维护      | 需要ReplacingMergeTree + 延迟 | PK Table 原生 Upsert       |
| 回测实盘统一      | 快表慢表分离，手动合并        | Union Read 一个 SQL        |
| 列裁剪            | 支持（查询时）                | 支持（服务端，网络传输层） |
| 极速路径（<50ms） | 不适合                        | 也不适合，走内存引擎       |

**架构图**：

```
CEX WebSocket ──┬──→ 内存规则引擎(C++/Rust) → CCXT API → 交易所  [极速层 <50ms]
                │
                ├──→ Fluss ──→ Flink ──→ 信号/风控              [实时层 <500ms]
                │         │
                │         └──→ Tiering ──→ Paimon
                │                            │
                └──→ ClickHouse ←────────────┘  [历史层 秒-分钟]
                     (OLAP/批量分析/回测)    (Union Read 统一查询)
```

ClickHouse 继续做 OLAP 和批量分析，Fluss 补流式关联和实时状态，两者通过 Paimon 共享历史数据。不是替换，是补位。"

## 关键话术备忘

### 要说的

- "ClickHouse 在你们这个规模下用得很好"——先承认
- "这不是 ClickHouse 的问题，是架构层的补位需求"——定位为补位
- "淘宝实测数据是……但你们场景可能不一样，需要实测验证"——诚实

### 不要说的

- "ClickHouse 不适合实时场景"——他说不定用得挺好
- "Fluss 可以替代你们的 Kafka + ClickHouse"——这是要推翻他

- "这个我确实没深入研究，但我可以回去查一下 Fluss 的文档和社区讨论，给你一个靠谱的回答"——诚实比编造好

## 关于这次沟通的核心心态

这次沟通跟上次跟 Darren 聊不一样。Darren 是老板，他看方向和战略；数据同事是工程师，他看可行性和细节。

**Darren 关心的是"你能帮我赚多少钱"，数据同事关心的是"你能不能帮我解决具体的工程问题"。**

所以这次要：

- 少讲愿景，多讲工程
- 少讲 Fluss 特性，多讲他的痛点

- 每个技术点都有具体的他们场景下的例子

**目标不是让他觉得 Fluss 很牛，而是让他觉得"这个人理解我的问题，而且可能有一个有用的解法"。**
