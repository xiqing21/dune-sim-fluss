# Fluss 架构在量化交易场景的适用性分析

## 回答核心问题：Fluss 是否太"后分析型"了？

---

## 报告摘要

Fluss 作为列式流存储系统，其设计目标是实时分析而非 T+1 报表。本报告从延迟、吞吐、功能三个维度分析 Fluss 在量化交易场景的适用性，结论是：对于 Spread 套利、Funding Rate 套利、链上 Alpha 信号、预言机偏差套利等秒级延迟策略，Fluss 完全适配；对于做市和抢跑等微秒级策略，Fluss 不适配但可以通过分层架构与极速决策层互补。Fluss 的核心价值不在于"比 Kafka 更快"，而在于"比 Kafka 更适合分析型量化场景"——列裁剪减少 80% 网络传输、主键 Upsert 免去下游去重、Delta Join 消除 TB 级状态、湖流一体 Union Read 消除回测与实盘偏差。

---

## 一、Fluss 的延迟特性深度分析

### 1.1 Fluss 不是"后分析型"系统

"后分析型"这个标签容易让人联想到 T+1 数据仓库、批处理 ETL、OLAP 报表。Fluss 的定位完全不同——它是**面向实时分析的列式流存储**，核心设计目标是亚秒级数据新鲜度。

Fluss 的存储架构是三层设计：

| 层次 | 介质 | 延迟 | 用途 |
|------|------|------|------|
| 热存储 | 本地磁盘（SSD） | < 1ms | 最新数据，实时读写 |
| 温存储 | 远程对象存储 | ~100ms | 近期数据，Flink 读取 |
| 冷存储 | Paimon 数据湖 | ~1s | 历史数据，批查询 |

量化交易的核心路径（信号生成、风控计算）访问的是热存储层，延迟亚毫秒级。这与 Kafka 的本地存储读取延迟（也是亚毫秒级）在同一量级。

### 1.2 与 Kafka 的延迟对比

很多人认为"Kafka 更快"，但实际上在分析型场景中，Fluss 可能比 Kafka 更快：

**场景：读取 Swap 事件的 price 字段做价差计算**

Kafka 方案：消费者读取整条消息（20+ 字段全量传输）→ 应用层解析 JSON/Avro → 提取 price 字段 → 计算。网络传输量大，CPU 解析开销高。

Fluss 方案：客户端只请求 price + amount + timestamp 3 个字段 → Fluss 服务端列裁剪，只返回请求的列 → 应用层直接使用。网络传输减少 80%，CPU 解析开销大幅降低。

**实测数据**（淘宝案例）：Swap 事件包含 20+ 字段但价格计算只需 5-6 个，Fluss 服务端列裁剪后，读取吞吐提升 10 倍，网络成本降低 70-90%。

### 1.3 Flink + Fluss 端到端延迟拆解

以 Spread 套利信号生成为例：

```
[CEX WebSocket 推送深度更新]
    │
    ▼ 延迟: 0ms（交易所推送）
[自定义 Flink Source 接收]
    │
    ▼ 延迟: ~5ms（网络+反序列化）
[写入 Fluss cex.depth_updates]
    │
    ▼ 延迟: ~10ms（Fluss 写入）
[Flink 作业读取 + Delta Join]
    │
    ├── Fluss 读取 onchain.dex_swaps: < 1ms（点查 KV Tablet）
    ├── Fluss 读取 cex.depth_updates: < 1ms（本地热数据）
    ├── 计算 Spread: < 1ms
    │
    ▼ 延迟: ~5ms
[写入 Fluss analytics.arb_signals]
    │
    ▼ 延迟: ~10ms
[策略引擎读取信号]
    │
    ▼ 延迟: < 1ms（Fluss 点查）
[CCXT API 下单]
    │
    ▼ 延迟: ~50-200ms（网络延迟为主）

端到端: CEX 推送 → 下单 ≈ 80-230ms
```

**关键发现**：Fluss + Flink 的内部处理延迟约 30ms，占总端到端延迟的 15-40%。瓶颈不在 Fluss/Flink，而在交易所的网络延迟和下单延迟。

## 二、量化策略的延迟需求分层

### 2.1 不是所有量化都需要微秒级

量化交易不是铁板一块，不同策略的延迟需求差异巨大：

| 策略类型 | 延迟需求 | 价差持续时间 | Fluss 适配性 |
|---------|---------|------------|-------------|
| 跨所 Spread 套利 | 100ms-1s | 分钟到小时 | 完全适配 |
| Funding Rate 套利 | 分钟级 | 小时到天 | 完全适配 |
| DEX-CEX 套利 | 1-5s | 秒到分钟 | 完全适配 |
| 链上 Alpha 信号 | 1-12s | 分钟到小时 | 完全适配 |
| 预言机偏差套利 | 1-5s | 秒到分钟 | 完全适配 |
| 清算预警 | 5-30s | 秒到分钟 | 完全适配 |
| DAO 治理攻击检测 | 5-30s | 分钟 | 完全适配 |
| 做市策略 | 1-10ms | 毫秒 | 部分适配（中低延迟走实时层，低延迟走极速层） |
| 抢跑/Sniper | < 1ms | 微秒 | 不适配 |
| 统计套利（HFT） | < 100us | 微秒 | 不适配 |

ForthTech 的主力策略（复合套利、Spread 套利、Funding 套利、DEX 套利、配对交易）全部在前 7 个策略范围内，Fluss 完全适配。同时 ForthTech 也在做市和流动性提供，中低延迟做市场景 Fluss 部分适配，低延迟做市需要走极速层。ForthTech 正在招聘 HFT 方向的 Quant Developer，三层架构可以很好地适配这个演进方向。

值得注意的是，根据 Darren 在 Web3 Festival 的分享，ForthTech 已经构建了虚拟订单簿系统，实时接入 30+ 家交易所，每秒处理 3 万行数据。这说明 ForthTech 在行情聚合层已有深厚积累，Fluss 的定位不是替代现有系统，而是**补齐跨源实时关联、风控信号统一分发、回测实盘零偏差**这些数据层能力缺口。

### 2.2 为什么 Spread 套利不需要微秒级

Spread 套利的核心逻辑是"现货 vs 合约价差"或"跨所价差"，价差的形成和消失是分钟级的：

- **Binance 合约 vs Gate 现货的 Spread**：由资金费率、市场情绪、流动性差异驱动，持续数小时到数天
- **DEX vs CEX 的价差**：由区块确认延迟（12s）驱动，持续秒到分钟
- **Funding Rate 套利**：每 8 小时结算一次，持仓周期通常是天级

微秒级延迟对这些策略没有意义——你不需要在 1 微秒内发现一个持续 30 分钟的 Spread。

## 三、Fluss 在量化场景的独特优势

### 3.1 列式流存储：减少 80% 无用数据传输

量化策略通常只关心少数几个字段：

| 数据类型 | 总字段数 | 策略所需字段 | 列裁剪收益 |
|---------|---------|------------|-----------|
| DEX Swap 事件 | 20+ | price, amount, reserve (5-6个) | 减少 70-80% 网络 I/O |
| CEX 深度更新 | 10+ | bid/ask top 5 (10个) | 减少 30-50% 网络 I/O |
| 链上 Transfer | 10+ | from, to, value (3个) | 减少 70% 网络 I/O |

Kafka 不支持列裁剪，消费者必须读取整条消息。在高吞吐场景（数百条链 × 每秒数千事件），这意味着大量无用的网络传输和 CPU 解析开销。

### 3.2 主键 Upsert：免去下游去重和状态维护

量化场景大量需要"实时更新"的数据：

- CEX 订单簿：每个价格档位的数量实时变化（`qty="0"` 表示撤单）
- 链上 Reorg：受影响区块的数据需要更新
- 持仓状态：每个仓位的保证金率实时变化
- Funding Rate：每个币种的费率每 8 小时更新

Kafka 只能追加写入，消费者需要自行去重和维护最新状态。Fluss 的 Primary Key Table 原生支持 Upsert——同一个主键的最新值自动覆盖旧值，消费者直接读取最新状态，无需去重。

**对量化的实际意义**：传统方案需要 Redis 或内存数据库来维护"当前状态"（订单簿、保证金率），Fluss 的 PK Table 可以替代这部分中间存储。

### 3.3 Delta Join：消除 TB 级 Join 状态

量化场景最频繁的操作是"价格与状态关联"：

- Swap 事件（价格流）+ 池子储备量（状态流）→ 计算实际滑点
- CEX 深度（价格流）+ 链上 Swap（价格流）→ 计算 Spread
- 清算事件（事件流）+ 用户持仓（状态流）→ 计算健康因子

传统 Flink 需要将右侧流的全量状态存入 RocksDB（可达 TB 级），检查点时间从秒级膨胀到分钟级。Fluss Delta Join 将状态外置到 Fluss 存储层——右侧流通过 Fluss 点查获取，Flink 作业零状态。

**淘宝实测数据**：双流 Join 状态从 100TB 降至 0，检查点时间从 90 秒降至 1 秒，资源消耗降低 10 倍（2300 CU → 200 CU）。

### 3.4 湖流一体 Union Read：消除回测与实盘偏差

量化策略最大的痛点之一是"回测与实盘不一致"。Lambda 架构下，回测用批处理数据（可能来自数据库），实盘用流处理数据（来自 WebSocket），两者格式、延迟、处理逻辑不一致，偏差率可达 20-30%。

Fluss + Paimon Union Read 解决了这个问题：

- 回测：Flink Batch 作业从 Paimon 读取历史数据，运行策略逻辑
- 实盘：同一个 Flink SQL 在流模式下运行，从 Fluss 读取实时数据
- **同一套 Flink SQL，同一个数据源（Fluss + Paimon），同一个处理逻辑**

这意味着策略开发者只需要写一遍代码，回测和实盘完全一致。

## 四、Fluss 在量化场景的局限性

### 4.1 不适合微秒级策略

Fluss 的热存储延迟是亚毫秒级，加上 Flink 处理延迟（约 5-20ms），端到端延迟在 30-200ms 量级。对于做市（需要 < 10ms）和抢跑（需要 < 1ms）策略，Fluss 太慢了。

但这些策略需要的是 FPGA/Co-location 级别的基础设施，Kafka 也同样不够快——这不是 Fluss 的问题，而是策略类型对基础设施的要求不同。

### 4.2 项目成熟度风险

Fluss 仍处于 Apache 孵化阶段（2024 年底开源，2025 年进入孵化），生产级 Web3 部署案例为零。Schema Evolution 仅支持 ADD COLUMN，KvTablet 不支持副本。

**应对策略**：核心交易路径（信号→执行）绕过 Fluss，直接走内存规则引擎→CCXT API；Fluss 负责风控、信号生成、历史分析等"非极速"路径。这样即使 Fluss 出问题，交易执行不受影响。

### 4.3 运维复杂度

自建 Fluss + Flink + Paimon 集群需要专业运维团队。对于中小型量化团队，这可能超出运维能力。

**应对策略**：优先使用云托管服务（阿里云 Flink/Ververica 已支持 Fluss），降低运维负担。

## 五、推荐架构：分层 + 混合

### 5.1 三层架构

```
极速决策层（交易执行路径，延迟 < 50ms）
┌──────────────────────────────────────────────┐
│ CEX WebSocket → 内存规则引擎(C++/Rust) → CCXT API → 交易所 │
│ 不经过 Fluss/Flink，直接内存处理                  │
│ 适合：HFT、低延迟做市、下单、止损、ADL 减仓等极速操作    │
│ 技术栈对齐：ForthTech 用 Python/C++/Rust，极速层天然匹配  │
└──────────────────────────────────────────────┘

实时分析层（风控 + 信号路径，延迟 < 500ms）
┌──────────────────────────────────────────────┐
│ CEX WebSocket → Fluss → Flink → 信号/风控       │
│ 提供 Delta Join、CEP、窗口计算等能力              │
│ 适合：Spread 信号、Funding 预警、链上 Alpha 监控   │
└──────────────────────────────────────────────┘

历史分析层（回测 + 研究路径，延迟秒级到分钟级）
┌──────────────────────────────────────────────┐
│ Fluss → Tiering → Paimon → Trino/StarRocks     │
│ 提供全量历史查询和复杂分析                         │
│ 适合：策略回测、绩效归因、新因子研究               │
└──────────────────────────────────────────────┘
```

### 5.2 数据流一致性保障

三层之间通过 Fluss 统一数据总线保持一致性：

- 极速层的执行结果回写到 Fluss（交易记录、仓位变化、PNL）
- 实时分析层从 Fluss 读取极速层的执行结果，更新风控状态
- 历史分析层通过 Tiering 自动获取全量数据

```
极速层执行结果 ──写入──→ Fluss ──Tiering──→ Paimon
                           │                    │
                    实时分析层读取         历史分析层查询
                    (更新风控状态)        (绩效归因/回测)
```

### 5.3 为什么这个架构比"全用 Fluss"或"全不用 Fluss"更好

"全用 Fluss"：交易执行路径延迟 80-230ms，对于 Spread 套利够用，但如果未来要做更快策略就受限。

"全不用 Fluss"：回到 FMZ/Hummingbot 的模式，风控逻辑每个策略重写一遍，回测与实盘割裂，数据管道碎片化。

"分层混合"：交易执行走极速路径（不依赖 Fluss），风控和信号走 Fluss 实时路径（利用 Fluss 的分析优势），回测走 Paimon 历史路径。三层共享数据，互不阻塞，渐进式升级。

## 六、结论

Fluss 不是"后分析型"系统，而是**面向实时分析的列式流存储**。它的延迟在亚毫秒级，与 Kafka 在同一量级，但在分析型场景中因为列裁剪、主键 Upsert、Delta Join 等特性，实际性能可能优于 Kafka。

对于 ForthTech 的主力策略（复合套利、Spread 套利、Funding 套利、DEX 套利、配对交易），Fluss 的延迟完全适配。对于做市和流动性提供，中低延迟场景 Fluss 部分适配，低延迟场景需要走极速层。对于正在招聘的 HFT 方向，三层架构天然支持——HFT 走极速层（C++/Rust 内存引擎），中性套利走实时层。

Fluss 在量化场景的核心价值不是"比 Kafka 更快"，而是"比 Kafka 更适合分析型量化"——减少 80% 无用数据传输、免去下游去重和状态维护、消除 TB 级 Join 状态、消除回测与实盘偏差。这些是传统消息队列无法提供的，但却是量化数据基础设施的核心需求。
